## Context

現在 `engine` クレートにゲームの主要なユースケースとドメインモデルが集約されており、データ永続化は `infrastructure` クレートに切り出されています。しかし、人間が直接ターミナル上でゲームをプレイできるフロントエンド環境が存在しません。デバッグやシステムテストのために、直感的に操作できるTUI（Text User Interface）アプリが必要です。

## Goals / Non-Goals

**Goals:**
- `ratatui` と `crossterm` を利用した TUI アプリケーションの構築。
- アプリの起動時に `infrastructure` のインメモリストレージ（またはファイルストレージ）を初期化し、`engine` のユースケースに注入するコンポジションルートの役割を持つこと。
- タイトル画面、大名選択画面、内政モード、戦争モードの画面遷移を管理する状態遷移ロジックの実装。
- ゲームの進行状況を画面中央にメッセージやログとして表示し、ユーザーが状況を把握できるようにする。

**Non-Goals:**
- ネットワーク通信を伴うマルチプレイの実装（今回はスタンドアロンのローカルプレイのみを対象とする）。
- グラフィックライブラリ（Bevy, SDLなど）を用いたGUIアプリケーションの提供。
- `engine` 側の既存ドメインロジックの破壊的変更（可能な限り既存APIを利用し、不足分のみRead Modelなどを追加）。

## Decisions

- **アーキテクチャ統合アプローチ**: 
  `cli` クレートに `App` 構造体を定義し、内部で現在の「画面の状態（Screen）」と「ゲームデータ（GameContext）」を保持します。ゲームデータ内には、初期化済みのユースケース（`DomesticUseCase` や `TurnProgressionUseCase` など）を持たせ、キー入力に応じて直接メソッドを呼び出します。
- **TUIライブラリ**: 
  最新のRust TUI標準である `ratatui` を採用し、ターミナル操作に `crossterm` を利用します。UIは各状態（タイトル、内政、戦争）ごとにレンダリング関数を切り出します。
- **状態管理**: 
  Enum `ScreenState` を用意し、`Title`, `SelectDaimyo`, `Domestic(CommandState)`, `War(WarState)` のように画面状態と付随する一時データ（カーソル位置や入力中の数値）を安全に保持します。

## Risks / Trade-offs

- **Risk**: コマンドの実行時に入力値（消費する金や米）を細かく指定するUIがあり、数値入力などのコンポーネントをターミナルで作り込むのに手間がかかる可能性がある。
  - **Mitigation**: 最初の段階では簡単なポップアップやメッセージプロンプトの形式で数値入力（数字キーでの単純入力や加減算）を受け付けるようにし、UIの実装コストを抑える。
- **Risk**: 既存の `engine` の Read側APIが不足しており、画面に描画すべき情報（領地ごとのパラメータなど）が取り出しにくい可能性がある。
  - **Mitigation**: 適宜 `engine` 側のリポジトリやFacadeに情報を取得するためのゲッターメソッドを追加することで対応する。
