## Context

現在 `engine` クレートにゲームの主要なユースケースとドメインモデルが集約されており、データ永続化は `infrastructure` クレートに切り出されています。しかし、人間が直接ターミナル上でゲームをプレイできるフロントエンド環境が存在しません。デバッグやシステムテストのために、直感的に操作できるTUI（Text User Interface）アプリが必要です。

## Goals / Non-Goals

**Goals:**
- `ratatui` と `crossterm` を利用した TUI アプリケーションの構築。
- アプリの起動時に `infrastructure` のインメモリストレージ（またはファイルストレージ）を初期化し、`engine` のユースケースに注入するコンポジションルートの役割を持つこと。
- タイトル画面、大名選択画面、内政モード、戦争モードの画面遷移を管理する状態遷移ロジックの実装。
- ゲームの進行状況を画面中央にメッセージやログとして表示し、ユーザーが状況を把握できるようにする。

**Non-Goals:**
- ネットワーク通信を伴うマルチプレイの実装（今回はスタンドアロンのローカルプレイのみを対象とする）。
- グラフィックライブラリ（Bevy, SDLなど）を用いたGUIアプリケーションの提供。
- `engine` 側の既存ドメインロジックの破壊的変更（可能な限り既存APIを利用し、不足分のみRead Modelなどを追加）。

## Decisions

- **アーキテクチャ統合アプローチ**: 
  `cli` クレートに `App` 構造体を定義し、内部で現在の「画面の状態（Screen）」と「ゲームの依存関係（UseCase等）」を保持します。インフラ層の初期化（マスターデータ読み込み）を行い、ユースケースを生成して `App` に注入します。キー入力に応じてこれらのユースケースを非同期に呼び出します。
- **TUIライブラリ**: 
  最新のRust TUI標準である `ratatui` を採用し、ターミナル操作に `crossterm` を利用します。UIは各状態（タイトル、内政、戦争）ごとにレンダリング関数を切り出し、`Frame` に対して描画を行います。
- **状態管理（ステートマシン）**: 
  Enum `ScreenState` を用意し、`Title`, `SelectDaimyo`, `Domestic`, `War` の遷移を管理します。さらに、各モード内で `DomesticSubState` や `WarSubState` を活用し、「数値入力中」「対象国選択中」「メッセージ表示中」といった一時的な状態を `Box<SubState>` による再帰構造などで柔軟に扱えるようにしました。
- **入力バリデーション**:
  プレイヤーの手番でない場合にはコマンド実行を拒否する `check_player_turn` ガードを実装し、ゲームの整合性を維持します。

## Risks / Trade-offs

- **Risk**: コマンドの実行時に入力値（消費する金や米）を細かく指定するUIがあり、数値入力などのコンポーネントをターミナルで作り込むのに手間がかかる可能性がある。
  - **Outcome**: `DomesticSubState::InputAmount` を導入し、数字キー入力を文字列としてバッファリングする簡素ながら強力な入力機構を実装することで対応しました。
- **Risk**: 既存の `engine` の Read側APIが不足しており、画面に描画すべき情報（領地ごとのパラメータなど）が取り出しにくい可能性がある。
  - **Outcome**: `UiSnapshot` 構造体を導入し、画面描画に必要な情報を一括で取得できるクエリ用ユースケースを `engine` に追加することで、UI層とドメイン層の分離を維持しつつ効率的な描画を実現しました。
