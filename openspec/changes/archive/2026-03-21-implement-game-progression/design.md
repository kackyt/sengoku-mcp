## Context
戦国統一ゲームはターン制シミュレーションであり、ターン内で各大名がランダムな順で行動します。現在の仕様は季節の変わり目のイベント計算のみで、「誰がいつ行動するか」のゲームステート管理がありません。

## Goals / Non-Goals
- Goals: 大名の行動順序の決定、全CPUでの自動行動（戦争処理含む）の実現、ターン進行に伴うイベント通知の実装と状態の永続化
- Non-Goals: この段階でのプレイヤー入力待ちや対話的なコマンド実行の構築（一旦全自動CPUで駆動させる）。CPUの高度な思考ルーチン（AI）の深い実装（簡易な自動行動にとどめる）。

## Decisions
- Decision: ドメインモデルに `GameState`（現在のターン数、現在の行動大名ID、行動済みの状態などを管理）を導入する。
  - Why: オニオンアーキテクチャ/DDDにおいて、現在のゲーム状況（誰の番か）を永続化・復元するためには、独立した状態管理モデルを定義するのが適しているため。
- Decision: ゲーム進行や行動結果をイベントとして定義し、UseCaseからインフラストラクチャ層等（MCPサーバへの通知など）へ送信可能にする。
  - Why: ターンが全自動で進むため、外部（クライアント等）がどのタイミングで誰が何をしたのかを推し量るにはイベント通知が不可欠なため。

## Risks / Trade-offs
- Risk: 既存の `Kuni` や `TurnService` の処理との一貫性。
  - Mitigation: `GameStateRepository` と `KuniRepository` を利用し、ユースケース内でまとめてトランザクション的に整合をとる手法を検討する。
