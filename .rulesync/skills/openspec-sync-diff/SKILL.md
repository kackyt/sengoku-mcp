# OpenSpec差分同期スキル (openspec-sync-diff)

このスキルは、コードの実装中に仕様スコープが拡大または変更された際、`main` ブランチと現在のブランチの差分を分析し、OpenSpecの各種ドキュメントを最新の実装に合わせて自動的に更新・検証する手順を提供します。

## 作業手順

以下のステップに従って、実装済みの機能とOpenSpec仕様書の同期を行ってください。

### 1. 差分の確認

まず、`main` ブランチと現在のブランチ（HEAD）間の変更内容を確認します。

- ターミナルで `git diff main...HEAD --stat` などのコマンドを実行し、変更されたファイル群を特定します。
- 差分出力から、追加または変更されたコードの中身を確認し、当初の `proposal.md` や `tasks.md` に記載されていない「新しいスコープ（要件・機能・修正）」を抽出します。

### 2. 対象となるOpenSpecの特定

現在作業中のOpenSpecの変更ディレクトリ（例: `openspec/changes/<change-name>/`）を特定します。

### 3. ドキュメントの更新

抽出した新しいスコープに基づいて、以下のOpenSpecドキュメントを更新します。

- **`proposal.md`**: `What Changes`（変更内容）や `Capabilities`（機能要件）のセクションに、追加で実装した機能や修正内容のスコープを追記します。
- **`design.md`**: 追加されたスコープに関する設計判断（Decisions）や目標（Goals）を追記します。
- **`tasks.md`**: 追加で行った作業をタスクリスト（チェックリスト）として末尾などに追記し、完了状態（`[x]`）にします。

### 4. Specファイル（仕様要件）の追加と修正

`Capabilities` に新しい項目を追加した場合、それに対応する仕様要件ファイルを `specs/` ディレクトリ内に作成、または既存のものを更新する必要があります。

- 新規のCapability（例: `cli-combat-validation`）ごとに `specs/<capability-name>/` ディレクトリを作成し、その中に `spec.md` を作成します。
- **仕様（ScenarioやRequirement等）は必ず日本語で記述してください。** 技術用語やプログラムを除き英語で記述してはいけません。
- **重要**: Fission-AI仕様に従い、各 `spec.md` の要求事項（Requirement）の要約文には、必ず RFC-2119 のキーワードである `MUST` または `SHALL` を含める必要があります。MUSTやSHALLは文頭に記述してください。
  - 例: `MUST: システムは、ターゲットの選択フローを必須で提供する。` や `SHALL: エンジンは、HPが0になったユニットをただちにデスポーンさせる。`
- 既存の `spec.md` が英語になっている場合や、不備（`MUST` や `SHALL` の欠如）がある場合も、この段階で日本語へ修正・翻訳しつつキーワードを補完します。

### 5. バリデーションの実行と確認

更新が完了したら、OpenSpecのバリデーションコマンドを実行して構文や整合性をチェックします。

- アプリケーションルートディレクトリで `pnpm exec openspec validate --all` を実行します。
- エラーが発生した場合は、CLIから出力されたエラーメッセージを確認し（必要に応じて `--type change --json <change-name>` を使用）、`SHALL/MUST` の不足などの構文エラーを修正して再度検証します。
- `Totals: X passed, 0 failed` となるまで修正を繰り返します。検証が通過すれば、差分の同期作業は完了です。
